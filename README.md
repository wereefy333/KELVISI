# KELVISI - Luxury Salon Management System

Полнофункциональная веб-система управления премиальным салоном красоты:
- публичный сайт с онлайн-записью;
- подтверждение записи через email-ссылку;
- кабинет мастера;
- административная CRM-панель;
- сервер API на Node.js + Express + Prisma + PostgreSQL.

Документ предназначен как техническая база для дипломной работы и описывает проект максимально подробно: от архитектуры до пользовательских сценариев и API-контрактов.

## Краткое описание проекта

Проект реализует единое пространство для трех зон:

- **Клиентская зона (`/`)**
  - презентационный лендинг салона;
  - многошаговая запись на услуги;
  - выбор одного или нескольких мастеров;
  - расчет доступных слотов с учетом занятости и графика;
  - отправка email для подтверждения;
  - переход на страницу подтверждения.

- **Кабинет мастера (`/barber`)**
  - просмотр расписания по дням/неделям;
  - фильтрация записей по статусу и поиску;
  - оперативная смена статусов (`IN_PROGRESS`, `COMPLETED`, `NO_SHOW`);
  - KPI мастера по выручке/конверсии/неявкам;
  - редактирование профиля (bio/experience/languages).

- **Админ-панель (`/admin`)**
  - сводка по дню, график выручки;
  - управление услугами;
  - управление мастерами и связанными аккаунтами;
  - управление статусами бронирований;
  - модерация отзывов;
  - CRM-клиенты, аккаунты пользователей, лист ожидания.

## Бизнес-цели и роли пользователей

### Бизнес-цели

1. Автоматизировать онлайн-запись без ручного колл-центра.
2. Снизить количество фейковых заявок за счет email-подтверждения.
3. Дать мастерам рабочий инструмент для управления текущим потоком клиентов.
4. Дать администратору полноценную операционную панель.
5. Централизовать данные: услуги, мастера, записи, отзывы, CRM-клиенты.

### Роли

- `CLIENT`:
  - создает запись через публичный сайт;
  - подтверждает запись по email-ссылке.

- `MASTER`:
  - входит в `/barber`;
  - управляет статусами своих записей;
  - ведет собственный профиль.

- `ADMIN`:
  - входит в `/admin`;
  - управляет справочниками и операционными сущностями;
  - модерирует отзывы и контролирует загрузку.

## Архитектура системы

Проект разделен на SPA-клиент и API-сервер.

```text
Browser (React SPA)
   |
   | HTTP /api/*
   v
Express API (Node.js)
   |
   | Prisma Client
   v
PostgreSQL
   |
   +--> SMTP (Nodemailer) -> Email with confirmation link
```

### Ключевые архитектурные особенности

1. **Frontend и backend отделены**
   - фронт рендерится Vite/SPA;
   - API работает на порту `3001`.

2. **Логика записи и слоттинга серверная**
   - сервер валидирует время;
   - сервер пересчитывает стоимость с учетом `priceMultiplier` мастера;
   - сервер проверяет конфликты по времени;
   - создание записи выполняется в транзакции PostgreSQL с защитой от гонки двойного бронирования.

3. **Email-подтверждение через токен**
   - запись создается в статусе `PENDING_EMAIL`;
   - клиент подтверждает через `/confirm?token=...`;
   - сервер переводит запись в `CONFIRMED`.

4. **Ролевая авторизация на API (HttpOnly Cookies)**
   - `POST /api/auth/login` выдает подписанный токен и устанавливает его в защищенную `HttpOnly` cookie.
   - `GET /api/auth/me` восстанавливает сессию при перезагрузке приложения.
   - Запросы к API используют `credentials: 'include'`.
   - `ADMIN` и `MASTER` получают только разрешенные им данные и действия.

5. **Авто-отмена неподтвержденных заявок**
   - фоновой задачей каждые 15 минут;
   - отмена `PENDING_EMAIL`, если старше 24 часов;
   - если SMTP-отправка не удалась, запись немедленно переводится в `CANCELLED`, чтобы слот не оставался занятым.

### Текущие меры безопасности

- приватные API проверяют токен из `req.cookies.token` или хедера `Authorization`;
- авторизация основана на защищенных `HttpOnly` куках для предотвращения XSS атак;
- роли `ADMIN` и `MASTER` разделены на уровне сервера;
- пароли в БД хранятся в виде `scrypt`-хеша;
- SMTP-секреты не хранятся в исходном коде и задаются только через env;
- `Master` связан с аккаунтом `User` через явный FK `userId`;
- создание записи защищено от race condition через транзакцию PostgreSQL и advisory lock.

## Подробное описание страниц и экранов

## Страница `/` - `ClientBooking.tsx`

Экран объединяет лендинг и wizard записи.

### Шаги мастера записи

Переменная `step` может принимать:

- `HOME`
- `GENDER`
- `SERVICE`
- `MASTER`
- `DATETIME`
- `FORM`
- `SUCCESS`
- `WAITLIST` (зарезервировано, отдельный рендер не реализован)

### Шаг `HOME` (лендинг)

Содержимое:

1. Hero-блок:
   - бренд `KELVISI`;
   - CTA-кнопка `Записаться Онлайн`.
2. Философия салона.
3. KPI-витрина (год основания, число клиентов, число мастеров, рейтинг).
4. Превью VIP-услуг (мужской/женский зал).
5. Секция мастеров.
6. Портфолио работ.
7. Секция преимуществ (стандарты сервиса).
8. Галерея салона (`SalonGallery`).
9. Бегущая строка отзывов.
10. Контакты + карта.
11. Footer с навигацией и ссылками.
12. Мобильная floating-кнопка записи.

### Шаг `GENDER`

- выбор `Мужской зал` или `Женский зал`;
- при выборе происходит сброс предыдущих selections;
- переход на шаг `SERVICE`.

### Шаг `SERVICE`

Функции:

- отображает услуги выбранной категории;
- позволяет выбрать несколько услуг;
- считает:
  - суммарную стоимость `totalPrice`;
  - длительность `totalDuration`;
  - в параллельном режиме `parallelDuration`.

Если услуг более одной:
- открывается модальное окно выбора режима:
  - `SEQUENTIAL` (последовательно);
  - `PARALLEL` (в 4 руки / параллельно).

### Шаг `MASTER`

Для каждой выбранной услуги:

1. выбор основного мастера:
   - конкретный мастер;
   - или `Любой свободный мастер`.
2. при необходимости выбор второго мастера (`secondaryMasterIds`).

Ограничения кандидатов задаются:

- через `SERVICE_MASTER_RULES` (жесткая матрица id в UI);
- fallback через `allowedMasterIds` в `Service`.

Кнопка `Далее` активна только когда:
- выбран основной мастер для каждой услуги;
- выбран вторичный мастер для обязательных услуг.

### Шаг `DATETIME`

Функции:

1. календарь по месяцам:
   - запрет выбора прошедших дат;
   - выбор даты в ISO `YYYY-MM-DD`.
2. динамический запрос свободных слотов:
   - `POST /api/booking-slots`;
   - учитывает режим выполнения и связку мастеров.
3. fallback-режим:
   - при сбое API строит локальные слоты;
   - выводит ошибку пользователю.
4. выбор конкретного времени `selectedTime`.
5. визуализация планов по услугам:
   - либо серверный `plan`;
   - либо fallback-план на клиенте.

### Шаг `FORM`

Блоки:

1. итог визита:
   - услуги;
   - режим;
   - мастера;
   - дата/время;
   - итоговая сумма.
2. форма данных клиента:
   - имя;
   - телефон;
   - email (обязателен).
3. отправка:
   - `POST /api/bookings`;
   - таймаут 20 секунд;
   - обработка ошибок SMTP/API.

### Шаг `SUCCESS`

- сообщает об отправке письма;
- выводит email получателя;
- подсказывает, что нужно нажать ссылку в письме;
- кнопка возврата на главную.

### Progress bar wizard

Показывается на шагах `GENDER`, `SERVICE`, `MASTER`, `DATETIME`, `FORM`:

- этапы: `Зал -> Услуги -> Мастер -> Время -> Запись`;
- допускается возврат на предыдущие пройденные шаги.

## Страница `/confirm` - `ConfirmBooking.tsx`

Логика:

1. читает `token` из query-параметра;
2. вызывает `GET /api/confirm-booking?token=...`;
3. показывает одно из состояний:
   - `loading`;
   - `success`;
   - `already` (уже подтверждена ранее);
   - `error`.

На успехе выводит карточку визита:
- услуга;
- дата;
- время;
- итоговая стоимость.

## Страница `/barber` - `Login` + `StylistCabinet`

### `Login` для мастера

- поля email/пароль;
- локальный показ демо-учеток;
- вызов `POST /api/auth/login` с `role: MASTER`;
- при успехе переход в кабинет мастера.

### `StylistCabinet`

#### Верхний уровень

- карточка мастера (фото/имя/роль);
- кнопка выхода.

#### KPI-блоки

- рейтинг;
- средний чек;
- конверсия;
- доля неявок;
- выручка за период.

#### Напоминания

- блок клиентов, чья запись начинается в ближайшие 30 минут.

#### Фильтры расписания

- режим просмотра: `day` / `week`;
- смещение даты вперед/назад;
- фильтр статуса;
- поиск по имени/телефону;
- диапазон дат `dateFrom/dateTo`.

#### Список записей

Показывается по дням в периоде, каждая запись содержит:
- время;
- статус;
- тайминг-бейдж (в ожидании/через N минут/опаздывает);
- клиент, телефон, услуги, сумма;
- заметки;
- действия:
  - `Начать` -> `IN_PROGRESS`;
  - `Завершить` -> `COMPLETED`;
  - `Неявка` -> `NO_SHOW`.

#### Профиль мастера

- просмотр/редактирование `bio`, `experience`, `languages`;
- сохранение в БД через `onUpdateMaster`.

#### Отзывы

- только `APPROVED` отзывы, привязанные к `bookingId` мастера.

## Страница `/admin` - `Login` + `AdminDashboard`

### `Login` для администратора

- аналогичный экран входа;

### `AdminDashboard` - вкладки

Вкладки:
- `dashboard`
- `bookings`
- `services`
- `masters`
- `reviews`
- `clients`
- `users`
- `waitlist`

### Вкладка `dashboard`

Содержит:

1. KPI карточки:
   - выручка за сегодня;
   - записи за сегодня;
   - средний чек;
   - отзывы на модерации.

2. График выручки по часам (Recharts).

3. Сетка занятости мастеров по часам.

4. Алерт "неактивные клиенты > 60 дней".

5. Таблица последних записей.

### Вкладка `bookings`

- таблица всех записей;
- поиск по имени клиента;
- inline-редактирование статуса через select;
- просмотр суммы и атрибутов записи.

### Вкладка `services`

- группировка по категориям (MEN/WOMEN);
- отображение VIP-бейджа и активности;
- модалка добавления/редактирования;
- удаление услуги.

### Вкладка `masters`

- карточки мастеров;
- индикаторы уровня/специализации/множителя;
- кнопка редактирования открывает `MasterModal`.

`MasterModal` поддерживает:

1. Профиль мастера:
   - имя, роль, фото, bio, опыт, языки;
   - рейтинг, уровень, множитель, активность;
   - специализация.

2. Аккаунт входа мастера:
   - email;
   - телефон;
   - пароль (создание или смена).

3. График работы:
   - по дням недели;
   - вкл/выкл дня;
   - время начала/конца.

4. Сохранение/удаление мастера.

### Вкладка `reviews`

- список отзывов со статусами;
- модерация:
  - `Approve` -> `APPROVED`;
  - `Reject` -> `REJECTED`.

### Вкладка `clients`

- CRM-таблица клиентов:
  - контакты;
  - визиты;
  - суммарный чек;
  - дата последнего визита;
  - заметки.

- подсветка неактивных клиентов.

### Вкладка `users`

- список аккаунтов `User`;
- роли и статусы активности;
- визуальные действия (редактировать/блокировать), в текущей версии без полноценного backend-CRUD для user-management.

### Вкладка `waitlist`

- лист ожидания:
  - клиент;
  - мастер;
  - предпочтительные даты;
  - кнопка "уведомить" (UI-действие).

## Сервер API (подробно)

Сервер находится в `server/server.mjs`.

### Базовые настройки

- `PORT` (по умолчанию `3001`)

### Ключевые валидации и бизнес-правила

#### `/api/bookings`

1. Требуется `clientEmail`.
2. Дата/время должны быть валидными.
3. Нельзя создавать запись в прошлое.
4. Мастер должен существовать и быть активным.
5. Стоимость пересчитывается на сервере:
   - сумма услуг из БД;
   - умножается на `master.priceMultiplier`.
6. Проверяется конфликт по времени:
   - сравнение интервалов с существующими активными бронями.
7. При успехе:
   - создается `confirmToken`;
   - статус `PENDING_EMAIL`;
   - отправляется письмо со ссылкой.

#### `/api/confirm-booking`

1. Токен обязателен.
2. Если токен не найден -> `404`.
3. Если запись отменена/завершена -> `410`.
4. Иначе статус устанавливается в `CONFIRMED`.

#### `/api/booking-slots`

1. Проверяет `date` и массив `services`.
2. Загружает мастеров и существующие брони.
3. Строит интервалы занятости по каждому мастеру.
4. Вычисляет доступные стартовые времена шагом 15 минут.
5. Возвращает до 40 вариантов слотов, каждый со структурой `plan`.

### Фоновая задача

Каждые 15 минут:

- ищет записи со статусом `PENDING_EMAIL`, созданные более 24 часов назад;
- массово переводит их в `CANCELLED`.

## Алгоритм подбора слотов и мастеров

Алгоритм реализован функциями:

- `buildSequentialPlan`
- `buildParallelPlan`
- `pickMaster`
- `isMasterFree`
- `getMasterWorkWindow`

### Входные параметры

- дата;
- список услуг (длительность + кандидаты мастеров);
- режим (`SEQUENTIAL` / `PARALLEL`);
- существующие записи мастеров;
- рабочие окна по `workSchedule`.

### Последовательный режим

- услуги выполняются одна за другой;
- следующая услуга начинается после окончания предыдущей;
- мастера резервируются локально в рамках расчета плана.

### Параллельный режим

- все выбранные услуги стартуют одновременно;
- длительность слота = максимум длительностей услуг;
- для каждой услуги подбирается основной (и опционально вторичный) мастер.

### Ограничения

- мастер должен быть в своем рабочем окне;
- пересечения интервалов не допускаются;
- второй мастер не может совпадать с первым в рамках одной услуги.

## Сквозной сценарий записи с email-подтверждением

1. Клиент на `/` выбирает:
   - зал -> услуги -> мастеров -> дату/время;
   - заполняет имя/телефон/email.
2. Клиент отправляет форму.
3. API:
   - валидирует данные;
   - проверяет конфликт;
   - создает `Booking(status=PENDING_EMAIL, confirmToken=uuid)`;
   - отправляет email с `APP_PUBLIC_URL/confirm?token=...`.
4. Клиент кликает ссылку из письма.
5. `ConfirmBooking` вызывает API подтверждения.
6. API устанавливает `status=CONFIRMED`.
7. Запись становится рабочей для админа/мастера.

## Управление состоянием и синхронизация с API

### Общий подход в `App.tsx`

1. На старте грузит только публичные данные: `services`, `masters`, `reviews`.
2. До ответа API использует fallback из `data.ts` для публичной части.
3. После входа в систему дополнительно подгружает приватные данные по роли:
   - `MASTER`: свои `bookings`;
   - `ADMIN`: `bookings`, `reviews`, `clients`, `waitlist`, `users`.
4. Токены `ADMIN` и `MASTER` хранятся в `sessionStorage` и отправляются в `Authorization`.

### Обновления данных

- `services`/`masters`/`reviews`/`bookings` обновляются через обработчики в `App.tsx`.
- `onUpdateBooking`, `onUpdateMaster` и др. пробрасываются вниз как props.

### Важные нюансы

- `/api/bookings` на сервере скрывает `PENDING_EMAIL`.
- публичный клиент не загружает CRM-данные и аккаунты пользователей.
- `MASTER` видит только собственные записи и не имеет доступа к CRM-спискам админа.

1. Создать запись на мастера в конкретный слот.
2. Повторно попытаться создать пересекающуюся запись.
3. Ожидать ответ `409` и сообщение о занятом времени.

1. Войти в `/barber` под MASTER.
2. Отфильтровать записи.
3. Перевести запись в `IN_PROGRESS` -> `COMPLETED`.
4. Проверить обновление KPI.

## D. Админка

1. Войти в `/admin`.
2. Добавить услугу.
3. Добавить мастера с аккаунтом.
4. Изменить статус записи.
5. Одобрить/отклонить отзыв.

## E. Авто-отмена неподтвержденных заявок

1. Создать запись и не подтверждать.
2. По истечении TTL убедиться, что статус стал `CANCELLED`.

## F. Ошибка SMTP

1. Временно отключить SMTP-конфигурацию или указать неверные `MAIL_*`.
2. Попробовать создать запись.
3. Ожидать понятную ошибку API.
4. Убедиться, что слот не остался занятым и запись не зависла в `PENDING_EMAIL`.

## G. Проверка RBAC

1. Открыть админские или мастерские API в Postman без токена.
2. Ожидать `401`.
3. Выполнить вход под `MASTER` и проверить, что доступны только свои записи.
4. Выполнить вход под `ADMIN` и проверить доступ к `clients`, `waitlist`, `users`.

## Известные ограничения и технический долг

1. Используется собственный подписанный токен API, а не промышленный стек вроде JWT refresh/cookies/Redis-session.
2. Хеширование паролей реализовано через `scrypt`, но без отдельного rate limit/lockout по попыткам входа.
3. Нет полного CRUD и серверных мутаций для вкладки `users` (часть действий UI-only).
4. Нет встроенных автотестов (unit/integration/e2e) в репозитории.
5. `MasterSchedule.tsx` выглядит как альтернативный/legacy-компонент и не подключен к текущему роутингу.
6. Тип шага `WAITLIST` в клиентском wizard объявлен, но отдельная ветка рендера не используется.
7. Часть PII по-прежнему доступна администратору в CRM-интерфейсе, что нормально для роли `ADMIN`, но требует аккуратной эксплуатации.

### 3. Исправления в backend и API

В серверной части были внесены изменения, необходимые для стабильной работы системы:

- добавлены новые маршруты для административных клиентских действий;
- исправлена работа advisory lock для PostgreSQL при создании записи;
- обновлена обработка waitlist с учетом архивирования;
- выполнена синхронизация Prisma schema с текущими бизнес-требованиями;
- устранены ошибки, связанные с несовпадением структуры базы и кода после переноса.

### 4. Настройка email-подтверждения записи

Была полностью настроена отправка писем подтверждения записи через SMTP:

- подключен отдельный почтовый ящик бота рассылки;
- SMTP-конфигурация вынесена в переменные окружения;
- переменные `MAIL_HOST`, `MAIL_PORT`, `MAIL_SECURE`, `MAIL_USER`, `MAIL_PASS`, `MAIL_FROM` переданы в Docker-контейнер backend;
- проверена фактическая отправка тестового письма;
- исправлено формирование письма подтверждения.

Отдельно доработан шаблон письма:

- письмо переведено в более аккуратный production-вид;
- добавлена текстовая версия письма (`text`) для совместимости с почтовыми клиентами;
- добавлен HTML-шаблон с корректным `UTF-8`;
- добавлены имя клиента, услуга, мастер, дата, время и стоимость;
- ссылка подтверждения формируется из `APP_PUBLIC_URL`, а не из локального адреса.

### 5. Подготовка production-стенда на Cloud.ru

Для демонстрации проекта по публичному IP был развернут production-стенд на виртуальной машине Cloud.ru.

Выполненные шаги:

- создана виртуальная машина с публичным IP;
- подключение к серверу настроено по SSH-ключу;
- установлен Docker и Docker Compose;
- проект склонирован на сервер из GitHub через SSH-доступ;
- настроен production `.env`;
- приложение запущено через `docker compose up -d --build`.

### 6. Перенос базы данных на сервер

Для сохранения реальных данных проекта была выполнена миграция базы на production-стенд:

- дамп базы данных `KELVISI.sql` перенесен с ноутбука на виртуальную машину;
- дамп был восстановлен в PostgreSQL внутри Docker-контейнера;
- после восстановления была выполнена адаптация схемы под текущую версию проекта;
- подтверждена работоспособность API после восстановления данных.

Во время этой части работы были решены прикладные проблемы совместимости:

- различие версий `pg_restore` между PostgreSQL 18 и PostgreSQL 16;
- конфликт уже существующих объектов базы;
- несовпадение структуры дампа и актуальной Prisma schema.

### 7. Настройка сетевого доступа на Cloud.ru

Для публикации проекта в интернете были выполнены сетевые настройки:

- найден и настроен раздел `Группы безопасности` в интерфейсе Cloud.ru;
- открыт входящий трафик по `22/tcp` для SSH;
- открыт входящий трафик по `80/tcp` для HTTP-доступа к сайту;
- проверена доступность приложения по публичному IP.

Отдельно было установлено, что:

- база данных наружу не опубликована;
- PostgreSQL доступен только внутри Docker-сети;
- это соответствует безопасному минимальному сценарию для демонстрационного стенда.

### 8. Выявленные и устраненные эксплуатационные проблемы

В ходе развертывания были найдены и решены реальные инфраструктурные ошибки:

- `SMTP disabled`, вызванный отсутствием передачи `MAIL_*` переменных в контейнер backend;
- `localhost` в ссылке подтверждения из-за использования локального `APP_PUBLIC_URL`;
- `502/504` при обращении к API после пересоздания контейнера `app`;
- зависимость `nginx` от старого внутреннего IP backend-контейнера;
- отсутствие части полей/таблиц после восстановления дампа;
- ошибки прав доступа на приватный SSH-ключ в Windows PowerShell.

Это важно для диплома, потому что показывает не только реализацию функционала, но и практическую эксплуатацию системы в реальном окружении.
