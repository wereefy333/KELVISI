# LUMIERE - Luxury Salon Management System

Полнофункциональная веб-система управления премиальным салоном красоты:
- публичный сайт с онлайн-записью;
- подтверждение записи через email-ссылку;
- кабинет мастера;
- административная CRM-панель;
- сервер API на Node.js + Express + Prisma + PostgreSQL.

Документ предназначен как техническая база для дипломной работы и описывает проект максимально подробно: от архитектуры до пользовательских сценариев и API-контрактов.

## Краткое описание проекта

Проект реализует единое пространство для трех зон:

- **Клиентская зона (`/`)**
  - презентационный лендинг салона;
  - многошаговая запись на услуги;
  - выбор одного или нескольких мастеров;
  - расчет доступных слотов с учетом занятости и графика;
  - отправка email для подтверждения;
  - переход на страницу подтверждения.

- **Кабинет мастера (`/barber`)**
  - просмотр расписания по дням/неделям;
  - фильтрация записей по статусу и поиску;
  - оперативная смена статусов (`IN_PROGRESS`, `COMPLETED`, `NO_SHOW`);
  - KPI мастера по выручке/конверсии/неявкам;
  - редактирование профиля (bio/experience/languages).

- **Админ-панель (`/admin`)**
  - сводка по дню, график выручки;
  - управление услугами;
  - управление мастерами и связанными аккаунтами;
  - управление статусами бронирований;
  - модерация отзывов;
  - CRM-клиенты, аккаунты пользователей, лист ожидания.

## Бизнес-цели и роли пользователей

### Бизнес-цели

1. Автоматизировать онлайн-запись без ручного колл-центра.
2. Снизить количество фейковых заявок за счет email-подтверждения.
3. Дать мастерам рабочий инструмент для управления текущим потоком клиентов.
4. Дать администратору полноценную операционную панель.
5. Централизовать данные: услуги, мастера, записи, отзывы, CRM-клиенты.

### Роли

- `CLIENT`:
  - создает запись через публичный сайт;
  - подтверждает запись по email-ссылке.

- `MASTER`:
  - входит в `/barber`;
  - управляет статусами своих записей;
  - ведет собственный профиль.

- `ADMIN`:
  - входит в `/admin`;
  - управляет справочниками и операционными сущностями;
  - модерирует отзывы и контролирует загрузку.

## Архитектура системы

Проект разделен на SPA-клиент и API-сервер.

```text
Browser (React SPA)
   |
   | HTTP /api/*
   v
Express API (Node.js)
   |
   | Prisma Client
   v
PostgreSQL
   |
   +--> SMTP (Nodemailer) -> Email with confirmation link
```

### Ключевые архитектурные особенности

1. **Frontend и backend отделены**
   - фронт рендерится Vite/SPA;
   - API работает на порту `3001`.

2. **Логика записи и слоттинга серверная**
   - сервер валидирует время;
   - сервер пересчитывает стоимость с учетом `priceMultiplier` мастера;
   - сервер проверяет конфликты по времени.

3. **Email-подтверждение через токен**
   - запись создается в статусе `PENDING_EMAIL`;
   - клиент подтверждает через `/confirm?token=...`;
   - сервер переводит запись в `CONFIRMED`.

4. **Авто-отмена неподтвержденных заявок**
   - фоновой задачей каждые 15 минут;
   - отмена `PENDING_EMAIL`, если старше 24 часов.

## Подробное описание страниц и экранов

## Страница `/` - `ClientBooking.tsx`

Экран объединяет лендинг и wizard записи.

### Шаги мастера записи

Переменная `step` может принимать:

- `HOME`
- `GENDER`
- `SERVICE`
- `MASTER`
- `DATETIME`
- `FORM`
- `SUCCESS`
- `WAITLIST` (зарезервировано, отдельный рендер не реализован)

### Шаг `HOME` (лендинг)

Содержимое:

1. Hero-блок:
   - бренд `LUMIÈRE`;
   - CTA-кнопка `Записаться Онлайн`.
2. Философия салона.
3. KPI-витрина (год основания, число клиентов, число мастеров, рейтинг).
4. Превью VIP-услуг (мужской/женский зал).
5. Секция мастеров.
6. Портфолио работ.
7. Секция преимуществ (стандарты сервиса).
8. Галерея салона (`SalonGallery`).
9. Бегущая строка отзывов.
10. Контакты + карта.
11. Footer с навигацией и ссылками.
12. Мобильная floating-кнопка записи.

### Шаг `GENDER`

- выбор `Мужской зал` или `Женский зал`;
- при выборе происходит сброс предыдущих selections;
- переход на шаг `SERVICE`.

### Шаг `SERVICE`

Функции:

- отображает услуги выбранной категории;
- позволяет выбрать несколько услуг;
- считает:
  - суммарную стоимость `totalPrice`;
  - длительность `totalDuration`;
  - в параллельном режиме `parallelDuration`.

Если услуг более одной:
- открывается модальное окно выбора режима:
  - `SEQUENTIAL` (последовательно);
  - `PARALLEL` (в 4 руки / параллельно).

### Шаг `MASTER`

Для каждой выбранной услуги:

1. выбор основного мастера:
   - конкретный мастер;
   - или `Любой свободный мастер`.
2. при необходимости выбор второго мастера (`secondaryMasterIds`).

Ограничения кандидатов задаются:

- через `SERVICE_MASTER_RULES` (жесткая матрица id в UI);
- fallback через `allowedMasterIds` в `Service`.

Кнопка `Далее` активна только когда:
- выбран основной мастер для каждой услуги;
- выбран вторичный мастер для обязательных услуг.

### Шаг `DATETIME`

Функции:

1. календарь по месяцам:
   - запрет выбора прошедших дат;
   - выбор даты в ISO `YYYY-MM-DD`.
2. динамический запрос свободных слотов:
   - `POST /api/booking-slots`;
   - учитывает режим выполнения и связку мастеров.
3. fallback-режим:
   - при сбое API строит локальные слоты;
   - выводит ошибку пользователю.
4. выбор конкретного времени `selectedTime`.
5. визуализация планов по услугам:
   - либо серверный `plan`;
   - либо fallback-план на клиенте.

### Шаг `FORM`

Блоки:

1. итог визита:
   - услуги;
   - режим;
   - мастера;
   - дата/время;
   - итоговая сумма.
2. форма данных клиента:
   - имя;
   - телефон;
   - email (обязателен).
3. отправка:
   - `POST /api/bookings`;
   - таймаут 20 секунд;
   - обработка ошибок SMTP/API.

### Шаг `SUCCESS`

- сообщает об отправке письма;
- выводит email получателя;
- подсказывает, что нужно нажать ссылку в письме;
- кнопка возврата на главную.

### Progress bar wizard

Показывается на шагах `GENDER`, `SERVICE`, `MASTER`, `DATETIME`, `FORM`:

- этапы: `Зал -> Услуги -> Мастер -> Время -> Запись`;
- допускается возврат на предыдущие пройденные шаги.

## Страница `/confirm` - `ConfirmBooking.tsx`

Логика:

1. читает `token` из query-параметра;
2. вызывает `GET /api/confirm-booking?token=...`;
3. показывает одно из состояний:
   - `loading`;
   - `success`;
   - `already` (уже подтверждена ранее);
   - `error`.

На успехе выводит карточку визита:
- услуга;
- дата;
- время;
- итоговая стоимость.

## Страница `/barber` - `Login` + `StylistCabinet`

### `Login` для мастера

- поля email/пароль;
- локальный показ демо-учеток;
- вызов `POST /api/auth/login` с `role: MASTER`;
- при успехе переход в кабинет мастера.

### `StylistCabinet`

#### Верхний уровень

- карточка мастера (фото/имя/роль);
- кнопка выхода.

#### KPI-блоки

- рейтинг;
- средний чек;
- конверсия;
- доля неявок;
- выручка за период.

#### Напоминания

- блок клиентов, чья запись начинается в ближайшие 30 минут.

#### Фильтры расписания

- режим просмотра: `day` / `week`;
- смещение даты вперед/назад;
- фильтр статуса;
- поиск по имени/телефону;
- диапазон дат `dateFrom/dateTo`.

#### Список записей

Показывается по дням в периоде, каждая запись содержит:
- время;
- статус;
- тайминг-бейдж (в ожидании/через N минут/опаздывает);
- клиент, телефон, услуги, сумма;
- заметки;
- действия:
  - `Начать` -> `IN_PROGRESS`;
  - `Завершить` -> `COMPLETED`;
  - `Неявка` -> `NO_SHOW`.

#### Профиль мастера

- просмотр/редактирование `bio`, `experience`, `languages`;
- сохранение в БД через `onUpdateMaster`.

#### Отзывы

- только `APPROVED` отзывы, привязанные к `bookingId` мастера.

- карточки мастеров;

2. Аккаунт входа мастера:
   - email;

   - пароль (создание или смена).

3. График работы:
   - по дням недели;
   - вкл/выкл дня;
   - время начала/конца.

4. Сохранение/удаление мастера.

### Вкладка `reviews`

- список отзывов со статусами;
- модерация:
  - `Approve` -> `APPROVED`;
  - `Reject` -> `REJECTED`.

### Вкладка `clients`

- CRM-таблица клиентов:
  - контакты;
  - визиты;
  - суммарный чек;
  - дата последнего визита;
  - заметки.

- подсветка неактивных клиентов.

### Вкладка `users`

- список аккаунтов `User`;
- роли и статусы активности;
- визуальные действия (редактировать/блокировать), в текущей версии без полноценного backend-CRUD для user-management.

### Вкладка `waitlist`

- лист ожидания:
  - клиент;
  - мастер;
  - предпочтительные даты;
  - кнопка "уведомить" (UI-действие).

## Сервер API (подробно)

Сервер находится в `server/server.mjs`.

### Базовые настройки

- `PORT` (по умолчанию `3001`)
- `HOST` (по умолчанию `0.0.0.0`)
- `APP_PUBLIC_URL` (используется для ссылки в email)
- `CORS_ORIGINS` (comma-separated)
- SMTP-параметры (`MAIL_*`)

### Список маршрутов

| Метод | URL | Назначение |
|---|---|---|
| `GET` | `/api/services` | Список услуг |
| `POST` | `/api/services` | Создание услуги |
| `PUT` | `/api/services/:id` | Обновление услуги |
| `DELETE` | `/api/services/:id` | Удаление услуги |
| `GET` | `/api/masters` | Список мастеров |
| `POST` | `/api/masters` | Создание мастера + user |
| `PUT` | `/api/masters/:id` | Обновление мастера + user |
| `DELETE` | `/api/masters/:id` | Удаление мастера (если нет связей) |
| `GET` | `/api/bookings` | Список записей (кроме `PENDING_EMAIL`) |
| `GET` | `/api/bookings/:id` | Получение записи |
| `PUT` | `/api/bookings/:id` | Обновление записи/статуса |
| `POST` | `/api/booking-slots` | Расчет доступных слотов |
| `POST` | `/api/bookings` | Создание записи + отправка email |
| `GET` | `/api/confirm-booking` | Подтверждение записи по токену |
| `GET` | `/api/reviews` | Список отзывов |
| `PUT` | `/api/reviews/:id` | Обновление статуса отзыва |
| `GET` | `/api/clients` | Список клиентов |
| `GET` | `/api/waitlist` | Список листа ожидания |
| `POST` | `/api/auth/login` | Авторизация |
| `GET` | `/api/users` | Список активных users |

### Ключевые валидации и бизнес-правила

#### `/api/bookings`

1. Требуется `clientEmail`.
2. Дата/время должны быть валидными.
3. Нельзя создавать запись в прошлое.
4. Мастер должен существовать и быть активным.
5. Стоимость пересчитывается на сервере:
   - сумма услуг из БД;
   - умножается на `master.priceMultiplier`.
6. Проверяется конфликт по времени:
   - сравнение интервалов с существующими активными бронями.
7. При успехе:
   - создается `confirmToken`;
   - статус `PENDING_EMAIL`;
   - отправляется письмо со ссылкой.

#### `/api/confirm-booking`

1. Токен обязателен.
2. Если токен не найден -> `404`.
3. Если запись отменена/завершена -> `410`.
4. Иначе статус устанавливается в `CONFIRMED`.

#### `/api/booking-slots`

1. Проверяет `date` и массив `services`.
2. Загружает мастеров и существующие брони.
3. Строит интервалы занятости по каждому мастеру.
4. Вычисляет доступные стартовые времена шагом 15 минут.
5. Возвращает до 40 вариантов слотов, каждый со структурой `plan`.

### Фоновая задача

Каждые 15 минут:

- ищет записи со статусом `PENDING_EMAIL`, созданные более 24 часов назад;
- массово переводит их в `CANCELLED`.

## Алгоритм подбора слотов и мастеров

Алгоритм реализован функциями:

- `buildSequentialPlan`
- `buildParallelPlan`
- `pickMaster`
- `isMasterFree`
- `getMasterWorkWindow`

### Входные параметры

- дата;
- список услуг (длительность + кандидаты мастеров);
- режим (`SEQUENTIAL` / `PARALLEL`);
- существующие записи мастеров;
- рабочие окна по `workSchedule`.

### Последовательный режим

- услуги выполняются одна за другой;
- следующая услуга начинается после окончания предыдущей;
- мастера резервируются локально в рамках расчета плана.

### Параллельный режим

- все выбранные услуги стартуют одновременно;
- длительность слота = максимум длительностей услуг;
- для каждой услуги подбирается основной (и опционально вторичный) мастер.

### Ограничения

- мастер должен быть в своем рабочем окне;
- пересечения интервалов не допускаются;
- второй мастер не может совпадать с первым в рамках одной услуги.

## Сквозной сценарий записи с email-подтверждением

1. Клиент на `/` выбирает:
   - зал -> услуги -> мастеров -> дату/время;
   - заполняет имя/телефон/email.
2. Клиент отправляет форму.
3. API:
   - валидирует данные;
   - проверяет конфликт;
   - создает `Booking(status=PENDING_EMAIL, confirmToken=uuid)`;
   - отправляет email с `APP_PUBLIC_URL/confirm?token=...`.
4. Клиент кликает ссылку из письма.
5. `ConfirmBooking` вызывает API подтверждения.
6. API устанавливает `status=CONFIRMED`.
7. Запись становится рабочей для админа/мастера.

## Управление состоянием и синхронизация с API

### Общий подход в `App.tsx`

1. На старте грузит данные через `Promise.all`.
2. До ответа API использует fallback из `data.ts`.
3. При ошибке API оставляет offline-данные и пишет warning в консоль.

### Обновления данных

- `services`/`masters`/`reviews`/`bookings` обновляются через обработчики в `App.tsx`.
- `onUpdateBooking`, `onUpdateMaster` и др. пробрасываются вниз как props.

### Важный нюанс

`/api/bookings` на сервере скрывает `PENDING_EMAIL`.
Это означает, что неподтвержденные email-заявки не возвращаются в общий список API до подтверждения.

## Известные ограничения и технический долг

1. Пароли пользователей хранятся в открытом виде (нет bcrypt/argon2).
2. Нет JWT/sessions и полноценной серверной авторизации по токену.
3. `Master` и `User` сейчас связываются по `name` в части сценариев, а не по жесткому FK.
4. Нет полного CRUD и серверных мутаций для вкладки `users` (часть действий UI-only).
5. Нет встроенных автотестов (unit/integration/e2e) в репозитории.
6. SMTP fallback-значения и секреты должны храниться только в env, не в коде/репозитории.
7. `MasterSchedule.tsx` выглядит как альтернативный/legacy-компонент и не подключен к текущему роутингу.
8. Тип шага `WAITLIST` в клиентском wizard объявлен, но отдельная ветка рендера не используется.
